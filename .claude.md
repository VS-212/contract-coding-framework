# Developer Instructions: Contract-Coding Workspace

You are developing inside a **Contract-Coding** workspace governed by a Single Source of Truth (SSOT) Language Contract. Follow these strict rules to maintain architectural integrity and prevent context bloat:

---

## 🧭 General Workflow

1. **Locate the Contract**: Find the Language Contract (usually `contract.yaml` or `api.contract.yaml`).
2. **Understand the Topology**: Look at the module definitions ($M_{i}$) and the topology graph (edges/dependencies).
3. **Respect Execution Layers**: Do not implement a module until all of its dependencies (nodes it points to) are fully implemented and verified.

---

## 🛠️ Code Generation Rules

* **Semantic Anchoring**: Every function or class you write MUST be annotated with its contract module ID using the following comment style directly above the declaration:
  ```python
  # @contract: M-YOUR-MODULE-ID
  def your_function():
      pass
  ```
* **Context Partitioning**: Do NOT read the entire source code of dependent modules unless absolutely necessary. Rely solely on the extracted signatures/skeletons of dependencies to avoid context rot and conserve context window efficiency.

---

## ⚙️ CLI Tools & Verification Gates

Use the following CLI commands (run via `uv run`) to check your work:

### 1. Autonomy Gate
Verify if your contract and environment are ready for autonomous execution:
```bash
uv run contract-coder check-autonomy contract.yaml
```

### 2. HEG Visualization
Visualize the dependency layout:
```bash
uv run contract-coder visualize contract.yaml
```

### 3. Structural Alignment Audit $E(\mathcal{C})$
Before declaring a task complete, run the AST alignment auditor:
```bash
uv run contract-coder audit contract.yaml src/
```
Ensure **Structural Integrity is 100.0%**. Any missing or forbidden edges will cause verification failure.
